# 第一讲 操作系统概述

单用户系统 -> 批处理系统 -> 多道程序系统 -> 分时系统
单体分层结构，从硬件驱动到用户界面，每一层只用低一层的功能。
微内核结构，它将操作系统的核心功能分解为多个相互独立的组件，一些高级功能，如文件系统、设备驱动程序和网络协议栈等则以用户空间进程的形式运行，尽可能把内核功能移到用户空间。- Minix, ANX
外核结构，操作系统的核心功能被设计成一组相对独立的外核，强调将操作系统的大部分功能封装在外核中，“外” 体现在与用户空间相对 “外”，对用户空间的功能提供了一个相对独立、抽象的接口。- Windows, MacOS, Linux, Android
虚拟机结构，将硬件资源抽象为虚拟机，每个虚拟机都是一个原始计算机系统的有效副本，并能完成所有的处理器指令

# 第一节 从 OS 角度看计算机系统

# OS 与硬件的关系

硬件 支持 OS 支持 应用
硬件 与 OS 的 边界 ：指令集 + 寄存器
OS 是对 硬件 的虚拟与抽象

# OS 与应用程序的关系

通过 系统调用 提供服务，进程的地址空间（memory layout）通过内核空间与用户空间不同的位置与权限界定了 OS/APP 的边界。

# 隔离机制

定义：操作系统中的应用程序不会影响（或破坏）其他应用或操作系统的正常执行或信息泄露
隔离需要建立边界
数据隔离：虚拟内存，若无许可，则每个程序无法访问不属于自己的内存；
控制隔离：特权模式，内核态可以执行用户态无法执行的特权操作
时间隔离：中断 / 异常机制

进程切换也是一种系统调用，通过 OS 内核切换进程保存上下文。

# 第二节 从 OS 角度看 RISC-V

# RISC-V 系统模式


ABI（Application Binary Interface）：应用程序二进制接口
SBI（Supervisor Binary Interface）：管理员二进制接口
SEE（Supervisor Execution Environment）：管理员执行环境
HAL（Hardware Abstraction Layer）：硬件抽象层
Hypervisor：虚拟机
当处理器执行当前特权模式不允许的操作时将产生一个异常，这些异常通常会产生自陷（trap）导致下层执行环境接管控制权

U-Mode：是应用程序运行的用户态 CPU 执行模式，不能执行特权指令，不能直接影响其他应用程序执行
S-Mode：限制 APP 的执行与内存访问，是操作系统运行的内核态 CPU 执行模式，直接影响应用程序
H-Mode：限制 OS 访问的内存空间，是虚拟机监控器的执行模式，直接影响 OS 运行
M-Mode：控制物理内存 / 直接关机，是 Bootloader/BIOS 运行的特权模式，直接影响所有软件执行

通过设置 CSR (控制状态寄存器) 实现隔离。

# RISC-V 系统编程

U-Mode 下的应用程序通过 ecall 进行环境调用（异常）进入操作系统访问物理资源
M-Mode 最重要的特性是拦截和处理中断 / 异常，包括设置中断标记，依据中断向量调用相应中断服务
RISC-V 提供一种中断 / 异常委托机制，通过该机制可以选择性地将中断 / 异常交给 S-Mode 处理，而完全绕过 M-Mode
虚拟地址将内存划分为固定大小的页来进行地址转换和内容保护

# 第四讲 多道程序与分时多任务

# 第一节 进程和进程模型

# 多道程序与协作式调度

多道程序：在内存中存在多个可执行程序，各个可执行程序共享处理器
协作式调度：可执行程序主动放弃处理器使用，而不是由操作系统打断

# 分时多任务与抢占式调度

分时多任务：各个可执行程序分时共享处理器，操作系统按时间片来给各个可执行程序分配 CPU 使用时间
进程：一个具有一定独立功能的程序在一个数据集合上的一次动态执行过程，也称为任务
抢占式调度：一种 “暂停 - 继续” 组合的执行过程，基于时钟硬件中断机制，操作系统可随时打断正在执行的程序

# 进程

特点：动态性，并发性，有限度的独立性
进程的组成包括程序、数据和进程控制块
进程控制块：操作系统管理控制进程运行所用的信息集合，与进程一一对应
创建 --> 就绪 --> 执行 --> 等待 --> 唤醒 --> 结束

# 第二节：实验

LibOS 支持在内存中驻留多个应用，形成多道程序操作系统
BatchOS 支持多道程序协作调度，应用程序主动放弃 CPU 并切换
任务上下文（Task Context）：应用运行在某一时刻的执行状态

任务切换不涉及特权级切换；Trap 切换涉及特权级切换；
任务切换只保存编译器约定的 callee 函数应该保存的部分寄存器；而 Trap 切换需要保存所有通用寄存器；
任务切换和 Trap 切换都是对应用是透明的；
Trap 切换需要硬件参与，任务切换完全由软件完成。

普通控制流是程序正常运行所遵循的流；异常控制流是执行环境发生切换的突变；控制流上下文是控制流在执行完某指令时的物理资源内容
包括函数调用上下文，中断异常上下文，任务上下文等

切换流程：

  1. 切换前，内核栈中存有任务 A 的 Trap 上下文和 Trap 处理函数的调用栈信息；
  2. A 在 A 的任务上下文里保存 ra/sp/ 通用寄存器快照；
  3. 读取 next_task_cx_ptr 指向的 B 任务上下文，恢复寄存器内容；
  4. 执行 ret 汇编指令，任务 B 从保存的 ra 寄存器处继续执行，此时 sp 已经指向内核栈中 B 的 Trap 上下文，实现控制流切换。

MultiprogOS 支持分时多任务 OS，通过设置时钟中断和抢占式调度实现

# 第五讲 物理内存管理

# 第一节 地址空间

逻辑地址 + 段式管理 --> 虚拟地址 (线性地址)
虚拟地址 + 页式管理 --> 物理地址
操作系统的内存管理高度依赖硬件中的 MMU

作用：运行程序直接实用虚存地址，不用关注数据存放具体位置；简化应用编译和加载运行；通过独立的地址空间来保护数据。

# 第二节 内存分配

运行应用所占内存按存储数据特征划分成多个段 (Segment)
静态内存分配：编译时的内存分配，包括全局、静态变量和代码，栈上
动态内存分配：运行时内存分配，显式（ malloc/free ）或隐式（垃圾收集），堆上

连续内存分配：容易发生内存碎片现象
动态分区分配：分配一个由进程指定，大小可变的分区（内存块），操作系统需要维护数据结构给已分配分区和空闲分区的状态
动态分区分配策略：

  • 最先匹配：最低地址空间开始
  • 最佳匹配：满足分配需求的最小分区
  • 最差匹配：满足分配需求的最大分区

伙伴系统：分配由小到大在空闲块中找最小可用块，如空闲块过大，对可用空闲块进行二等分，直到得到合适可用空闲块；合并满足大小相同2i2^{i}2i，地址相邻且低地址是2i+12^{i+1}2i+1 倍数

非连续内存分配：允许一个程序使用非连续的物理地址空间，允许共享代码与数据和动态加载链接
非连续分配的硬件辅助机制：
段式存储管理：内存中存有段表，每个段可能有不同的大小（代码段，数据段），由操作系统内核管理，对应任务进程；程序的逻辑地址通过段表（段号和偏移量）翻译成物理地址
页式存储管理：内存中存有页表，由操作系统内核管理，物理页面和逻辑页面的基本单位大小相同
段页式存储管理：每个段维护页表项

反置页表：将虚拟地址和物理地址之间的映射关系存储在一个全局的页表中，索引是物理地址，当需要进行地址翻译时，它会根据虚拟地址映射成哈希值查找全局页表，找到对应的物理地址和进程标识符。

# 第三节 实验

初始化 & 使能页机制：M 模式的 RustSBI 在第一次进入 S-Mode 之前会把 0 写入 satp ，以禁用分页；然后 S-Mode 的 OS 在初始化页表后会再次写 satp

# 用户视角的地址空间

一系列有关联和不一定连续的逻辑段，进程对代码和数据的直接访问范围限制在它关联的地址空间之内
跳板页：所有应用相同，映射到同一物理页，放置内容是 trap.S 中的代码
产生异常时，CPU 会跳转到跳板页的入口，执行保存相关寄存器到 TrapContext 操作，同时使用 sscratch 保存应用的 TrapContext 地址，并通过陷入上下文的寄存器获取页表基址 / 内核栈指针 / trap_handler 地址

# 内核管理地址空间

需要管理物理内存，将剩下的空闲内存以单个物理页帧为单位管理起来
建立和拆除虚实地址映射关系：在多级页表中找到一个虚拟地址对应的页表项；通过修改页表项的内容即可完成键值对的插入和删除，从而实现映射关系的建立和拆除
逻辑段：一段连续地址的虚拟内存 MapArea
地址空间：一系列有关联的不一定连续的逻辑段 MemorySet = MapArea + PageTable

# 第六讲 虚拟存储管理

# 第一节 虚拟存储概念

虚拟存储：操作系统以页为单位自动换入换出内存
覆盖技术：不同时间段内执行的函数或模块共享一块有限的空间
覆盖：把程序划分为一系列功能相对独立的程序段，让执行时不要求同时装入内存的程序段组成一组（称为覆盖段），共享主存的同一个区域；必要部分常驻内存，可选部分只在需要时装入内存，不存在调用关系的模块共用同一块内存
交换技术：操作系统以程序为单位自动换入换出内存；只当内存空间不够或有不够的可能时换出
虚拟存储技术的前提是程序具有局部性：时间，空间，分支
基本特征：不连续性，大用户空间，部分交换

在何处保存放在外存中的页的地址：交换空间或是磁盘上的文件

# 第二节 局部页面置换算法

页面置换的时机：空闲内存数量到达上限，暂停回收内存；到达下限，开始回收内存
页面锁定 / 常驻内存：必须常驻内存的逻辑页面，页面中设置锁定标志位
评价方法：更少的缺页，更好的性能
局部页面置换算法：置换页面的选择范围仅限于当前进程占用的物理页面内

# 最优页面置换算法 OPT

基本思路：置换在未来最长时间不访问的页面，缺页时计算内存中每个逻辑页面的下一次访问时间
但是实际系统中无法预知每个页面在下次访问前的等待时间，只能通过两次模拟同一个程序

# 先进先出页面置换算法 FIFO

基本思路：选择在内存驻留时间最长的页面进行置换，维护一个记录所有位于内存中的逻辑页面链表，链表元素按驻留内存的时间排序
性能较差，调出的页面可能是经常访问的。

# 最近最久未使用算法 LRU

基本思路：选择最长时间没有被引用的页面进行置换，缺页时，计算内存中每个逻辑页面的上一次访问时间
实现方式：

  1. 系统维护一个按最近一次访问时间排序的页面链表；访问内存时，找到相应页面，并把它移到链表之首；
  2. 维护活动页面栈，访问页面时，将此页号压入栈顶，并栈内相同的页号抽出；缺页时置换栈底的页面；
    两种方式的开销都很大。

# 时钟页面置换算法 Clock

在页表项中增加访问位，包括

  • 驻留位：1 内存；0 外存
  • 访问位：1 访问；0 未访问
  • 页帧号：物理页面位置
    置换流程：
  1. 各页面组织成环形链表，指针指向最先调入的页面，每次访问页面则访问位置 1；
  2. 缺页时，从指针当前位置顺序检查；若访问位为 0，则置换该页；否则访问位置 0，并将指针移动到下一个页面，直到找到可置换的页面；

# 改进的时钟页面置换算法

在页面中增加修改位，描述页面在过去一段时间的内写访问情况；记（访问位，修改位）为元组。
置换流程：

  1. 从当前位置开始扫描到第一个（0, 0）的帧用于替换，不修改任何标志位；
  2. 若第一轮扫描失败，则重新扫描，查找第一个（0, 1）的帧用于替换，本轮将所有扫描过的帧访问位设为 0；
  3. 若第二轮扫描失败，则重新扫描，查找第一个（0, 0）的帧用于替换，不修改任何标志位；
  4. 若第三轮扫描失败，则重新扫描，查找第一个（0, 1）的帧用于替换；

# 最不常用置换算法 LFU

基本思路：缺页时，置换访问次数最少的页面
每个页面设置一个访问计数，每次访问递增，开销较大；且开始时频繁使用，但以后不使用的页面很难置换，可以通过计数定期右移来解决

# Belady 现象

采用 FIFO 等算法时，可能出现分配的物理页面数增加，缺页次数反而升高的异常现象；因为被置换出去的页面并不一定是进程近期不会访问的
LRU 算法没有 Belady 现象；Clock 算法是二者的折衷，但它不能记录准确访问顺序，因此也会发生 Belady 现象

# 第三节 全局页面置换算法

思路：为进程分配可变数目的物理页面
运行程序少时，提高程序运行数，可提高 CPU 利用率；但程序运行数量大导致内存访问增加，并会降低访存的局部性，从而导致缺页率上升和 CPU 利用率下降

# 工作集页面置换算法

一个进程当前正在使用的逻辑页面集合，可表示为二元函数W(t,Δ)W(t, Δ)W(t,Δ)，其中ttt 为当前时刻，ΔΔΔ 为一个定长的页面访问时间窗口，其大小为τττ；工作集表示在当前时刻ttt 前的ΔΔΔ 时间窗口中的所有访问页面所组成的集合。
进程开始执行后，随着访问新页面逐步建立较稳定的工作集；当内存访问的局部性区域位置稳定时，工作集大小也大致稳定；局部性区域位置改变时，工作集快速扩张和收缩过渡到下一个稳定值
常驻集：当前时刻进程实际驻留内存中的页面集合，取决于系统分配给进程的物理页面数目和页面置换算法
当工作集是常驻集的子集时，缺页率较少；工作集发生剧烈变动时，缺页较多

算法思路：换出不在工作集中的页面；通过维护窗口内的访存页面链表实现。

# 缺页率置换算法

算法思路：通过调节常驻集大小，使每个进程的缺页率保持在一个合理的范围内
访存时，设置引用位标志；缺页时，计算从上次缺页时间tlastt_{last}tlast​ 到现在tcurrentt_{current}tcurrent​ 的时间间隔，如果超过可以容忍的缺页窗口，则置换所有在[tlast,tcurrent][t_{last}, t_{current}][tlast​,tcurrent​] 未被引用的页；否则增加缺失页到常驻集中

抖动问题：随着驻留内存的进程数目增加，分配给每个进程的物理页面数不断减小，缺页率不断上升；操作系统需在并发水平和缺页率之间达到一个平衡

# 第七讲 进程管理与单处理器调度

# 第一节 进程管理

# 进程的基本概念

引入进程的目的：清晰地刻画操作系统中程序运行的动态内在规律；有效管理和调度多个程序的执行和对资源的使用
进程提供给应用程序的抽象：独立的逻辑控制流，私有的地址空间
详细定义：一个具有一定独立功能的程序在某数据集合上的一次执行和资源使用的动态过程
从实现角度看，进程是操作系统建立程序运行中的过程管理相关的数据结构，以及对数据结构的动态操作过程
从资源角度看，进程是程序执行中占用资源的集合
进程可以在运行的过程中，创建子进程、用新的程序内容覆盖已有的程序内容，并成为程序执行过程中动态申请 / 使用 / 释放各种资源的载体，而任务不能

# 进程管理

fork() 复制父进程的所有变量，内存，CPU 寄存器（程序计数器 PC 除外）；子进程的 fork() 返回 0，父进程返回子进程的 pid ，如果出错则返回值为负数
进程控制块 PCB，用于负责进程的调度、切换、保存和恢复等工作。

wait() 系统调用用于父进程等待子进程的结束，有子进程存活时，父进程等待子进程的返回结果；某个子进程结束 exit() 时唤醒父进程，将其返回值作为 wait() 的返回值返回
僵尸进程：已经 exit() 但没有被父进程 wait() 回收 PCB 的子进程
孤儿进程：其父进程先退出的子进程，由 root 进程等待并回收

exit() 的作用：

  1. 关闭所有打开的文件等占用资源；释放内存；释放大部分进程相关的内核数据结构
  2. 保留结果的值，检查是否父进程存活；如没有存活，设置父进程为 root 进程
  3. 进入僵尸状态，等待父进程回收

其他： nice() 指定进程的初始优先级； ptrace() 允许一个进程控制另一个进程的执行； sleep() 可以让进程在定时器的等待队列中等待指定时间

# Fork

在 99% 的情况里，我们在调用 fork() 之后调用 exec() ，因此开销昂贵的内存复制是没有作用的；可以写时复制
vfork() 创建进程时，不再创建一个同样的内存映像；使用 COW 技术

# 第二节 单处理器调度

# 处理机调度概念

从就绪队列中挑选下一个占用 CPU 运行的进程，从多个可用 CPU 中挑选就绪进程可使用的 CPU 资源
调度时机：进程从运行状态切换到等待 / 就绪状态，或被终结，或主动放弃 CPU 时；以及中断请求被服务例程响应完成时（可抢占式）

比较调度算法的准则：
吞吐量与延迟准则：低延迟、高计算带宽，减少开销，减少就绪等待时间
响应时间准则：及时处理用户的输入请求，尽快将输出反馈给用户；减少平均响应时间的波动
公平准则：保证每个进程占用相同的 CPU 时间，保证每个进程的就绪等待时间相同

# 调度算法

先来先服务调度算法 (FCFS)：依据进程进入就绪状态的先后顺序排列；指标周转时间，表示所有进程从初始化到结束时间的平均；缺点是平均等待时间波动较大，I/O 资源和 CPU 资源的利用率较低

短作业优先调度算法 (SJF)：选择就绪队列中执行时间最短进程，特点是具有最优平均周转时间，但可能导致长作业饥饿，并且需要预知进程的执行时间，用历史的执行时间来预估（τn+1=αtn+(1−α)τnτ_{n+1}=αt_{n}+(1-α)τ_{n}τn+1​=αtn​+(1−α)τn​）

最短剩余时间算法 (SRT)：支持抢占调度机制，有新的进程就绪且服务时间小于当前进程的剩余时间，则转到新的进程执行

最高响应比优先算法 (HRRN)：对 FCFS 调度算法和 SJF 调度算法的平衡，计算后备作业队列中每个作业的响应比，选出最高的作业投入运行（响应比R=(w+s)/sR=(w+s)/sR=(w+s)/s，www 为就绪等待时间，sss 为执行时间）

时间片轮转算法 (RR)：按照时间片分配处理机资源；时间片结束时，按 FCFS 算法切换到下一个就绪进程
时间片太大：等待时间过长，退化成 FCFS
时间片太小：产生大量上下文切换开销，影响到系统吞吐量
经验规则：维持上下文切换开销处于 1% 以内

多级队列调度算法 (MQ)：就绪队列被划分成多个独立的子队列，每个队列具有同一优先级和自己的调度策略
队列间的调度：如果固定先处理优先级更高的，可能导致饥饿；可以通过时间片轮转，根据每个队列分配时间（80%-20%）

多级反馈队列调度算法 (MLFQ)：工作进入系统时，放在最高优先级；如在当前的时间片没有完成，则降到下一个优先级；如在其时间片内主动释放 CPU，则优先级不变；时间片大小随优先级级数增加（即优先级降低）而增加
特征：CPU 密集型进程的优先级下降很快，而 I/O 密集型进程停留在高优先级
经过一段时间 S，就将系统中所有工作重新加入最高优先级队列

公平共享调度算法 (FSS)：控制用户对系统资源的访问，按比例分配

# 第三节 实时调度

# 实时操作系统

专为实时应用程序设计的操作系统，性能指标重要的是时间约束的及时性，需要在严格的时间限制内完成任务，具有时间约束的可预测性
强 / 硬实时操作系统：必须在指定时间内完成重要的任务；弱 / 软实时操作系统：尽量但非必须完成
周期实时任务：周期 p = 任务请求时间间隔；执行时间 e = 最大执行时间
硬时限：错过任务时限会导致灾难性或非常严重的后果；必须验证，在最坏情况下能够满足时限
软时限：通常能满足任务时限
可调度表示一个实时操作系统能够满足任务时限要求，优先级调度包括静态和动态，以任务执行过程中是否会改变优先级区分

# 实时调度

速率单调调度算法 (RM)：根据任务周期来确定任务优先级，周期越短优先级越高，且为抢占式
进程 P1：e=25 p=50
进程 P2：e=35 p=80

最早截止时间优先算法 (EDF)：任务的优先级根据当前任务的截止时间动态分配，截止时间越短，优先级越高，抢占式
进程 P1：e=10 p=20
进程 P2：e=25 p=50

最低松弛度优先算法 (LLF)：根据任务紧急或者松弛程度，来确定任务优先级；松弛度 = 必须完成时间 - 本身还需要运行时间 - 当前时间

# 优先级反置

高优先级进程长时间等待低优先级进程所占用资源的现象：

优先级继承：占用资源的低优先级进程继承申请资源的高优先级进程的优先级
只在想占有资源的高优先级进程被阻塞时，才提高占有资源的低优先级进程的优先级

优先级天花板协议：占用资源进程的优先级与所有可能申请该资源的进程的最高优先级相同，从而避免了死锁和优先级反转等问题

# 第八讲 多处理器调度

# 第一节 对称多处理与多核架构

单核处理器
超线程处理器：在单个处理器核心上实现多个逻辑处理器，每个都能够独立地执行指令流，充分调动 CPU 内部闲置的处理资源
多核处理器：在同一处理器芯片上集成多个处理器核心的技术
众核处理器：在同一处理器芯片上集成了十几个甚至几百个处理器核心

对称多处理器（SMP）：每个处理器核心都可以访问共享的系统资源，访问延迟相同
非一致内存访问系统（NUMA）：内存分布在多个节点中，每个处理器核心只能直接访问其本地节点的内存，需要维护一致性

Cache 一致性问题：缓存写回脏数据的不一致问题，内存数据的版本一致性问题等

# 第二节 多处理器调度概述

# 单队列多处理器调度 SQMS

待执行的任务都排队在一个全局的任务队列中，每个处理器核心空闲后可以从任务队列中取出一个任务进行执行
多处理器调度的亲和度：尽可能让特定进程在同一个 CPU 上运行
这种方法缺乏缓存亲和性和可扩展性

# 多队列多处理器调度 MQMS

每个处理器核心都可以有自己的任务队列，任务根据其特性、优先级或其他属性被分配到不同的队列中
跨 CPU 迁移：不断地迁移和切换一个或多个进程，可实现负载均衡
工作窃取：进程量较少的队列不定期查看其他队列是否比自己多，如果更满则从目标窃取一个或多个进程
检查间隔：综合考虑开销和负载均衡

# 第三节 Linux O (1) 调度

Linux 1.2：环形队列 + Round Robin 调度策略
Linux 2.0：全局锁的单一控制权对内核访问串行化，在用户态支持并行，内核本身并不能利用多处理器加速
Linux 2.2：引入调度类，用于实现不同进程调度策略

Linux 2.4：O(n)O(n)O(n) 调度器，时间复杂度和活跃进程的数量成正比
思路：

  • 把时间分成大量的微小时间片，每次使用前检查所有就绪进程的优先级
  • 进程被调度器切换执行后，可不被打扰地用尽这个时间片
  • 如进程没有用尽时间片，则剩余时间增加到进程的下一个时间片中

只用一个 Global Runqueue 放置就绪任务，交由每个 Core 竞争；有大量进程运行时性能大大降低，扩展性差

Linux 2.6：O(1)O(1)O(1) 调度器

# 第九讲 文件系统

# 第一节 文件和文件系统

文件系统是存储设备上组织文件的方法和数据结构；是操作系统中负责文件命名、存储和检索的子系统。
所有一切均抽象成文件，提供了统一的接口
文件是具有符号名（标识符），由字节序列构成的数据项集合
文件头：文件系统元数据中的文件信息

文件视图：
用户：字节序列的集合
操作系统：数据块的集合（逻辑存储单元），存储在扇区中（物理存储单元）
文件系统中的基本操作单位是数据块，无论实际访问数据的大小都要获取整个数据块

文件访问控制：UNIX 模式，<拥有者用户 | 所属组 | 其他所有用户，r|w|x> r=4，w=2，x=1

操作系统为每个进程维护一个打开的文件描述符表，通过文件指针指向系统级的所有打开文件表；
打开文件表记录了当前文件指针的位置，文件状态标志（读写、阻塞、同步、附加），引用计数（归零时释放），i-node 指针，锁信息（可选）

文件系统是操作系统中管理持久性数据的子系统，提供数据文件命名、存储和检索功能
目录是一类特殊的文件，内容是文件索引表 <文件名，指向文件的指针>，应用程序通过系统调用对目录进行操作
目录的实现方式：线性表，哈希表

硬连接：多个文件项指向一个文件的 i-node，删除只有所有的硬连接被删除时才会删除数据
软连接：新建文件时以存储文件名称的方式来指向其他文件，而不是直接指向 i-node，可以指向目录或其他文件系统的文件，删除后变成悬垂连接

挂载：将一个文件系统集成到操作系统的目录树中，使其内容可以通过指定的目录路径进行访问（/mnt，mount 临时挂载点，/home，/usr，/var 持久挂载点，通过配置文件自动挂载）

# 第二节 文件系统的设计与实现

虚拟文件系统 VFS：定义了一组所有文件系统都支持的数据结构和标准接口，对接真实文件系统（如磁盘文件系统 XFS，Ext，网络文件系统 NFS 等）

基本的数据结构：
文件卷控制块（超级块 superblock）：包含文件系统类型、大小，数据块与 inode 块大小、总数、空闲数，根目录的 inode，最近一次挂载时间、写入时间、检查间隔等
文件控制块 inode：属于每一个文件，包含文件大小，数据块位置向量，访问模式，时间信息，拥有者或群组标识，链接数；对于目录的 inode 存有指向其目录项数据块的位置
Bitmap：inode 块和数据块使用或者未使用标志位
目录的数据块包含目录项，每个目录项包括文件（目录，如 ./，../）名和指向该文件或子目录的 inode 指针

数据块按需读入内存，在使用后被缓存；虚拟页式存储统一缓存数据块和内存页，将虚拟地址空间中的页面映射到外存

文件数据块的分配方式：连续分配（最先、最佳...），链式分配（显式包含指针，隐式维护外部结构 FAT 记录先后关系），索引分配（链式、多级）

空闲空间管理：使用位图，Di=0D_i=0Di​=0 代表空闲

文件读写创建操作流程：

多数磁盘划分为一个或多个分区，每个分区有一个独立的文件系统。

# 第三节 支持崩溃一致性的文件系统

崩溃可能导致磁盘文件系统映像中的文件系统数据结构出现不一致，同时更新磁盘上的两个结构，但在二者中途崩溃就会出现。
目标：将文件系统从一个一致状态原子地变迁到另一个一致状态

举例：将单个数据块附加到已有文件末尾
场景一：只将数据块写入磁盘，但没有指向它的 inode，好似从未发生
场景二：只将 inode 写入磁盘，读取的是垃圾数据
场景三：只将 Bitmap 写入磁盘，显示已经分配，导致文件系统永远不会使用这个块，空间泄露
场景四：写入 inode 和 Bitmap，垃圾数据
场景五：写入 inode 和数据块，版本不一致
场景六：写入 Bitmap 和数据块，没有指向它的 inode，版本不一致

文件系统检查程序 fsck：

  • 超级块检查：确保文件系统大小大于分配的块数，如果内容不合理可以决定使用备用副本
  • 位图与 inode 间的一致性检查：如果位图和 inode 之间存在任何不一致，通过信任 inode 内的信息来解决；确保看起来在用的 inode 都拥有标记
  • inode 状态检查：类型字段有效，难以修复时清除
  • 链接计数检查：扫描所有目录并统计每个 inode 实际引用次数，与 inode 记录的链接计数比较，不一致则修复；对于没有目录条目引用的 inode（孤立文件），将其移动到 lost+found 目录，为用户提供机会手动检查和恢复
  • 重复指针检查：两个不同的 inode 引用同一个块的情况
  • 坏块检查：如果指针显然指向超出其有效范围的某个指针，或者出现环状数据块链，则被清除
  • 目录检查：确保 "." 和 ".." 是前置条目，每个 inode 都已分配

缺点：效率很慢

日志文件系统：
预写日志：更新磁盘时，在覆写结构之前，首先写下一点小注记，描述将要做的事；发生崩溃时，能够返回并查看，然后重试
数据日志（Journal Mode）：包含三个阶段

  • 日志写入：将事务开始标记 TxB 和事务数据（元数据和数据）写入日志
  • 日志提交：将事务提交块（包含事务结束标记 TxE，校验和等）写入日志
  • 加检查点：将更新内容（元数据和数据）写入其最终的磁盘位置

崩溃恢复：如果在提交前崩溃则直接丢弃日志，如果在提交后检查前，则对所有提交的日志进行重放（replay）
缺点：太多写操作，慢

日志文件系统的性能优化：
日志超级块：单独区域存储，包含日志的首尾指针，记载序列号，完成循环日志回收与复用
元数据日志（Ordered Mode）：强制首先将数据块写入（不包括 Bitmap，inode），保证指针永远不会指向垃圾数据；日志中事务数据的数据块内容替换为元数据
另一种 Writeback Mode：直接写入元数据，速度最快

# 第十讲 进程间通信

# 第一节 进程间通信 (IPC) 概述

目标：多进程协作完成复杂应用需求
定义：进程间通过数据交换（共享或传递）进行交互的行为
例如 cat README.md | grep rcore

直接通信：不需要内核中转就能相互传递信息（共享内存）
间接通信：进程间通过系统调用和内核的中转（信号，管道，消息队列，Socket，文件）

阻塞通信：进行通信等待时阻塞进程，对应非阻塞通信
缓冲方式：无限容量，有限容量，0 容量（必须等待成功接收）

管道：有读写端的一定大小的字节队列，读写端通过不同的文件描述符表示，创建管道时添加入进程文件描述符表
应用场景：父子进程（有关系的进程）之间通信，通常管道两端的进程会各自关闭管道的一个文件描述符，固定读写功能

匿名管道（shell 中 | 连接符）只支持父子兄弟进程通信；命名管道（mkfifo）可以支持任意两个进程间的通信，是阻塞式的单向通信管道，表现为一个特殊的文件

消息队列：由操作系统维护的以结构数据为基本单位的间接通信机制，相同类型标识的消息组成按先进先出顺序组成一个消息队列
msgget(key, flags) ：创建一个新的消息队列或获取一个现有的消息队列标识符
msgsnd(qid, buf, size, flags) ：发送信息，消息类型蕴含在 buf 结构中
msgrcv(qid, buf, size, type, flags) ：接受信息， type=0 接收任意信息， type>0 接收指定信息， type<0 接收类型小于 abs(type) 的信息
msgctl() ：控制消息队列的操作，如删除消息队列、获取或设置属性
key 值可通过宏定义或者 ftok 函数生成（基于文件路径名和一个项目标识符）

共享内存：把同一个物理内存区域同时映射到多个进程的内存地址空间
shmget(key, size, flags) ：创建共享段
shmat(shmid, *shmaddr, flags) ：把共享段映射到进程地址空间
shmdt(*shmaddr) ：取消共享段到进程地址空间的映射
shmctl() ：控制信息

信号：中断正在运行进程的异步消息或事件
信号发送：进程（kill）及外设硬件（ctrl+c）通过内核发出信号，或者内核直接发出信号
信号接收：进程注册了信号的处理函数，可以选择忽略，捕获，或者使用内核默认的处理方式

# 第十一讲 线程

# 第一节 线程

进程之间交换数据不方便，地址空间相互隔离，管理开销大，难以处理并发并行问题
线程 vs 进程

  • 进程是资源（包括内存、打开的文件等）分配的单位，线程是 CPU 调度的单位；
  • 进程拥有一个完整的资源平台，而线程只独享必不可少的资源，如寄存器和栈；
  • 线程同样具有就绪、阻塞、执行三种基本状态，同样具有状态之间的转换关系；
  • 线程能减少并发执行的时间和空间开销，切换控制流轻量，管理便捷

线程是进程的一部分，是进程中的指令执行流的基本单元，是 CPU 调度的基本单位
进程 = 线程 + 共享资源

创建线程： int pthread_create(pthread_t * thread, const pthread_attr_t * attr, void * (*start_routine)(void*), void * arg);
阻塞当前线程，等待目标线程终止： int pthread_join(pthread_t thread, void **retval);

用户态管理且用户态运行的线程：由一组用户级的线程库函数来完成线程的管理，内核感知不到线程的存在；控制简单，开销少，灵活；但同一进程中只能同时有一个线程在运行，发生系统调用则阻塞整个进程，只能按进程分配 CPU 时间，不能基于线程抢占处理机
内核态管理且用户态运行的线程：由内核维护线程控制块 TCB，线程执行系统调用而被阻塞不影响其他线程（不占用其他线程的内核栈资源）；但线程切换开销大，多线程 fork() 会引起子进程内容不一致（竞态条件），死锁问题（父进程中的线程持有锁，子进程复制锁的状态为持有），开销更大
内核态管理且内核态运行的线程：线程在内核中执行，用户态感知不到该线程，如周期性的 Buffer-Cache 写回任务，虚存交换任务，文件系统的事务日志等
混合管理且运行的线程：轻量级进程 LWP 代表跟内核线程一对一映射，可以对应多个用户线程，用户和内核以 LWP 为单位协同管理；具有最大灵活度和实现复杂性；1:1，N:1，M:N（几个用户线程对应几个 LWP）

线程切换：只需要切换线程的私有数据、寄存器等不共享的数据

# 第十二讲 同步与互斥

# 第一节 概述

进 / 线程如果有资源共享，可能出现难以重现的错误，存在不确定性
原子操作是指一次不存在任何中断或失败的操作，操作系统需要利用同步机制保证一些操作是原子操作

进入区：检查可否进入临界区的一段代码，如果可以则设置正在访问标志
临界区：一段需要互斥执行的代码
退出区：清除正在访问标志
剩余区：其余部分

空闲则入：没有线程在临界区时，任何线程可进入
忙则等待：有线程在临界区时，其他线程均不能进入临界区
有限等待：等待进入临界区的线程不能无限期等待
让权等待：不能进入临界区的线程，应释放 CPU

同步互斥的方法：

  • 禁用硬件中断：进入临界区时禁止所有中断并保存标志，离开临界区时使能所有中断并回复标志；会导致禁用中断后线程无法停止，引起饥饿，临界区时间可能很长难以确定，要小心使用
  • 基于软件：都需要忙等待
    1. Peterson 算法：先举手示意，设置让他人优先，当他人没有举手且暂时确实不希望使用（则会让给自己优先）时使用
    1. Dekkers 算法：先举手示意，如果他人举手且轮到他人先放下手，等到不再轮到他人再次举手，访问完毕把轮次交给他人
    1. N 线程：共享 turn 变量，所有线程排列成环，有多个想进入临界区时从前往后走，执行完一个线程， turn 改为下一个线程的值；线程 Ti 要等待从 turn 到 i-1 的线程都退出才能访问
  • 更高级的抽象方法：操作系统提供更高级的编程抽象来简化线程同步

现代 CPU 都提供一些特殊的原子操作指令：Test-and-Set（读取值并设置为 1，返回原始值），Compare-and-Swap（比较当前值与预期值，相等则更新为新值并返回 1，否则返回 0）
ABA 问题：
线程 T1 读取变量 X，值为 A；
线程 T2 读取变量 X，值也为 A；
线程 T2 将变量 X 修改为 B，然后再次修改为 A，此时 T1 阻塞；
线程 T1 现在准备将 X 的值从 A 修改为 C，使用 CAS 指令进行原子操作。因为线程 T1 看到 X 仍然是 A，所以 CAS 操作成功，将 X 的值从 A 修改为 C。
从线程 T1 的角度来看 X 没有变化，但实际上它被 T2 修改过一次到 B。
解决方法：加上版本号，两次 X 的 A 值具有不同版本号，证明它被修改过。

锁：通过原子操作指令实现，忙等锁 vs 等待锁

适用于单处理器或者共享主存的多处理器中任意数量的线程同步，线程离开临界区时有多个等待线程的情况，可能导致饥饿

# 第二节 信号量

一种抽象数据类型，由一个整型 (sem) 变量和两个原子操作组成
P 操作： sem = sem - 1 ，若 sem < 0 进入等待
V 操作： sem = sem + 1 ，若 sem <= 0 代表有线程在等待，唤醒一个等待的

互斥访问：每个临界区设置一个初值为 1 的信号量（二进制信号量），成对使用 P 和 V 操作，避免重复遗漏
条件同步：每个条件同步设置一个初值为 0 的信号量，任何 P 操作需要等待 V 操作（放入资源）才能执行

生产者 - 消费者问题：任何时刻只能有一个线程操作缓冲区（互斥访问）；缓冲区空时，消费者必须等待生产者（条件同步）；缓冲区满时，生产者必须等待消费者（条件同步）
需要信号量 mutex, fullBuffers, emptyBuffers，无法避免死锁问题

# 第三节 管程与条件变量

传统 PV 和锁机制可读性差，程序正确性难以保证
管程：采用面向对象方法，任一时刻最多一个线程执行管程代码

管程是一种特殊的数据类型，其中不仅有数据，而且有对数据进行操作的方法；其中的某些过程实现了某些功能，在其外部则是不可见的
条件变量：一个结构体，包含了一个等待队列和一些基本操作函数；通过它实现线程的阻塞、唤醒和通讯等功能
条件变量与互斥锁配合，可实现线程间的同步和互斥；互斥锁用于保护共享资源，而条件变量则用于线程间的等待和通知机制
int pthread_cond_init(pthread_cond_t *cond, const pthread_condattr_t *attr); ：初始化
int pthread_cond_destroy(pthread_cond_t *cond); ：销毁
int pthread_cond_wait(pthread_cond_t *cond, pthread_mutex_t *mutex); ：将线程放入条件变量的等待队列，并解锁传入的互斥锁
int pthread_cond_signal(pthread_cond_t *cond); ：唤醒一个等待在条件变量上的线程
int pthread_cond_broadcast(pthread_cond_t *cond); ：唤醒所有等待在条件变量上的线程

管程的组成：一个锁，一些条件变量及其等待队列，入口等待队列，紧急等待队列

  • T.enter：进入管程之前要获得互斥锁访问权，如果成功获得则进入入口等待队列
  • T.leave：当线程离开管程时，如果紧急队列不为空，唤醒紧急队列中的线程，并将它所持锁赋予唤醒的线程；否则释放 lock，唤醒入口等待队列某个线程
  • T.wait (c)：阻塞自己并挂到 c 的等待队列，释放锁，唤醒入口等待队列的一个或多个线程
  • T.signal (c)：把 c 的等待队列某个线程唤醒，把自己的锁交给它（内部操作），自己挂到紧急等待队列

线程 T2 的 signal，使线程 T1 等待的条件满足时，实现方式有：

  • Hoare：T2 通知完 T1 后，T2 阻塞，T1 马上执行；等 T1 执行完，再唤醒 T2 执行
  • Hansen：T2 通知完 T1 后，T2 执行，执行结束后（规定以 signal 结束）离开管程，将锁直接交给 T1，后者执行
  • MESA：T2 通知完 T1 后，T2 还会接着执行，T1 并不会立即执行，而是重新竞争权限（回到入口队列，被调度时继续要检查条件变量）

根据哲学家座位的奇偶性，判断先取左边还是右边的筷子，可以避免死锁。
AND 型信号量集：在一个原语中申请整段代码需要的多个临界资源，要么全部分配给它，要么一个都不分配给它；使用 AND 型信号量集同时申请两边的筷子，可以避免死锁。

# 第五节 死锁

由于竞争资源或者通信关系，两个或更多线程在执行中出现，永远相互等待只能由其他进程引发的事件（线程 1 获得锁 1 希望获得锁 2，线程 2 获得锁 2 希望获得锁 1）
资源分类：

  • 可重用资源：任何时刻只能有一个线 / 进程使用资源，释放后可以重用
  • 可消耗资源：信号，消息等可被销毁的资源，进程间可能相互等待接受对方的消息，导致死锁

资源分配图：描述资源和进程间的分配和占用关系的有向图，以进程和资源为顶点，如果进程 P 请求资源 R 则连边 P->R，如果资源 R 分配给进程 P 则连边 R->P
存在死锁的必要条件：1. 锁必须互斥；2. 进程处于持有并等待资源的状态；3. 资源无法抢占只能资源释放；4. 资源分配图中存在等待进程集合之间存在相互等待资源的环

死锁预防：

  • 把互斥的共享资源封装成可同时访问，破坏互斥；
  • 只在能够同时获得所有需要资源时才分配，破坏持有并等待，资源利用率低；
  • OS 协助剥夺已占有资源，破坏非抢占，开销很大；
  • 对资源排序，要求进程按顺序请求资源，破坏循环等待，必须按规定难以变化；

死锁避免：在分配资源时判断是否会出现死锁，只在不会死锁时分配资源；要求进程声明需要资源的最大数目
安全状态：

  • 针对所有已占用进程，存在安全执行序列 <P1, P2, ..., Pn>
  • Pi 要求的资源 <= 当前可用资源 + 所有 Pj 持有资源，其中 j < i
  • 如 Pi 的资源请求不能立即分配，则 Pi 等待所有 Pj 完成，j < i
  • Pi 完成后，Pi+1 可得到所需资源，执行完并释放所分配的资源

系统处于安全状态，一定没有死锁
银行家算法：P729

死锁检测算法：P741

死锁恢复：终止所有死锁进程，或者选择进程对其资源进行抢占，进程回退到一些安全状态，可能导致同一进程一直被抢占
